Auth, Rôles & Onboarding

Deux rôles, un seul produit — ce que chacun voit

Flux d'authentification

1
Paul crée son compte manager
Email + mot de passe via Supabase Auth. Profile créé avec role=manager. Onboarding : saisir la date de début de commission (défaut : 01/01/2025).
2
Paul invite l'artiste par email
Depuis Réglages → "Inviter l'artiste". Supabase envoie un lien magique. À l'ouverture, le profile est créé automatiquement avec role=artist.
3
Artiste installe la PWA
Onboarding artiste : 1 écran "Ajoute l'app à ton écran d'accueil" avec instructions iOS/Android. Requis pour les push notifications. Pas bloquant — l'app fonctionne sans.
Sécurité — Supabase RLS
L'artiste ne peut lire que les colonnes exposées par des policies explicites. Toute la logique de permission est dans Postgres — pas dans le code React, qui peut être inspecté.

Policies RLS clés
payments : artiste voit uniquement SELECT sur ses propres lignes (artist_id = auth.uid())
management_fees : artiste voit uniquement SUM(commission_due) via une view dédiée — jamais la table directement
tasks : artiste voit uniquement WHERE assignee_role = 'artist'
grants, tracks, expenses : SELECT bloqué pour role=artist au niveau Postgres
